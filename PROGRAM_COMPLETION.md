# Sideways Program Completion

This document is the human-readable companion to `audit/release-completion.json` and `scripts/release-completion.mjs`. It does not replace executable evidence and it does not convert external infrastructure into repository truth.

## Demonstrated code-local completion

The final candidate is admitted only when `npm run verify:release` and `npm run verify:repository` prove all of the following on one exact tree:

- the root reader gives one ordinary-language promise joining public discovery to a person-owned private archive, with a direct `/manual/` action and ordinary recommendation explanations;
- private memory remains canonical in IndexedDB, supports immutable-generation OPFS mirrors, bounded or streaming Ark export, transactionally rollback-governed restore, quota/interruption failure receipts, and no server projection in the private Ark;
- every repository-defined consumer social operation is reachable by its exact HTTP method in relational mode, dangerous controls are absent without actor authority, and static/blob modes never simulate shared success;
- ranking evaluation covers eligibility, saturation hysteresis, posterior-family advantage, smoothed automatic gate, deep-saturation floor, explicit seeded exploration, greedy set diversity, matched candidate pools, replay instability, and the delayed-feedback privacy contract;
- operations have deterministic redacted snapshots, a read-only live probe, disposable-only migration/backup/restore/rollback evidence, named authorities, stop conditions, and exact rollback procedures;
- workflow permissions, authority mapping, path leases, supply-chain provenance, Maker boundaries, and repository verification remain fail-closed.

## Exact-head workflow admission

Code-local completion is not admitted until every workflow named in `audit/release-completion.json` passes against the exact PR head. This includes the repository gate, kernel parity, real Pages root build, root phone and desktop quality proof, manual/profile journey, social phone journey, Ark phone journey, media journey, authority assembly, workflow authority, Maker runtime, overlay validation, and lasso checks.

The browser evidence covers Chromium at 390×844 and desktop, keyboard focus and activation, 200% and 400% text reflow, computed contrast, reduced motion, offline and failed API behavior, storage/quota failures, zero unexpected page errors, and no horizontal overflow on the exercised primary journeys.

A workflow success from another commit, a stale artifact, or a synthetic source string is not admission evidence.

## Post-merge deployment proof

After the admitted PR merges, the final `main` commit must pass its required workflows and the live deployment must serve:

```text
/.well-known/sideways-deployment.json
```

The sentinel’s `commit` must equal the final merge commit and its `repository` must equal `Pokitomas/theawesomehexapp`. The open deployment receipt issue must be updated by the authorized Pages deployment to record that same served commit. A successful build without served-commit convergence is not deployment completion.

Optional Netlify functions and PostgreSQL are not claimed deployed unless a dated redacted operations snapshot and read-only probe prove them. Static Pages remains honest about unavailable shared mutation.

## External and intentionally unsupported boundaries

The final report keeps these boundaries explicit rather than guessing:

- conventional browser-download retention after a download request;
- device-specific browser and operating-system storage eviction;
- current Netlify function, PostgreSQL, backup/restore, branch/environment, application-grant, runner, rate-limit, and production-cookie configuration until dated evidence exists;
- screen reader journeys such as VoiceOver, TalkBack, or NVDA;
- Firefox and WebKit behavior beyond the current Chromium proof;
- representative-device startup, memory, and interaction latency budgets.

These boundaries do not disguise unresolved code-local gaps. They name observations the repository cannot truthfully manufacture.

## Termination

Once the final merge, exact `main` checks, live sentinel, deployment receipt, completion-issue closure, and lease cleanup are all proved:

1. close issues #257–#262 and #264 with exact commit and workflow receipts;
2. close superseded coordination or expired-agent tasks as replaced, not completed by those agents;
3. ensure no competing Sideways-completion `maker/*` or `agent/*` pull request remains open;
4. stop the Sideways completion swarm.

This termination applies only to the bounded Sideways release-completion campaign. It does not prohibit evidence-backed product work governed by [`PRODUCT_PORTFOLIO.md`](./PRODUCT_PORTFOLIO.md), Archie and Maker intelligence/execution research, independent Sideways retention work, or the Frontier World Expo campaign. Those programs must use their own exact promotion gates and may not reopen completed Sideways requirements merely to create motion.
