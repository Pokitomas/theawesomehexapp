# Parallel Runtime Directive — Game-Changing Native Model Foundry

## Mission

Build a native model-development system whose purpose is not to implement a predetermined Transformer, Mamba, mixture-of-experts, distillation, quantization, or agent architecture.

Its purpose is to repeatedly discover, test, falsify, combine, train, and retain model designs that move the empirical Pareto frontier:

- greater generalizable capability;
- less active compute;
- less memory traffic;
- less training data or training cost where possible;
- stronger adaptation and reliability;
- and an architecture or learning mechanism materially different from merely shrinking or routing an existing large language model.

The system must be capable of finding solutions the operators did not specify in advance.

## Governing rule

No agent may convert an attractive hypothesis into the project architecture without comparative evidence.

Mamba, attention, recurrence, state-space models, MoE, latent reasoning, test-time learning, retrieval, memory, novel optimizers, synthetic curricula, alternative tokenization, modularity, program induction, and external tools are candidate ingredients only.

The foundry owns the search space. Human suggestions seed experiments but do not define the answer.

## Parallel read-only roles

Run all roles concurrently. They may inspect supplied repository and research context, but they may not mutate files, install dependencies, spend training budget, or claim execution without evidence.

1. **Frontier Cartographer** — identify fixed, accidental, removable, and unexplored assumptions and the smallest discriminating experiments.
2. **Architecture Heretic** — generate structurally different model families; at least half must not be straightforward Transformer/Mamba/MoE hybrids.
3. **Learning-Dynamics Inventor** — search beyond next-token pretraining and ordinary fine-tuning; name the signal and exactly what state or parameters it changes.
4. **Distillation Breaker** — transfer algorithms, uncertainty, search policies, abstractions, memory policies, and failure recognition; identify routes for a student to exceed its teachers.
5. **Benchmark Saboteur** — create hidden and procedural tests for memorization, contamination, verbosity theater, prompt brittleness, tool-cost hiding, and inference-budget cheating.
6. **Efficiency Physicist** — measure active FLOPs, bytes moved, peak and resident memory, cache growth, latency, energy proxies, training cost, and local throughput.
7. **Mechanistic Pathologist** — diagnose dead capacity, state collapse, routing monoculture, shortcuts, unstable fast state, forgetting, bottlenecks, and removable components.
8. **Open-World Toolsmith** — identify pinned, licensed, sandboxed external libraries, repositories, datasets, simulators, compilers, kernels, and evaluators tied to named experiments.
9. **Scaling-Law Skeptic** — separate tiny-model artifacts, optimization advantages, data advantages, architectural advantages, and inference-budget advantages.
10. **Product Reality Agent** — define local launch, hardware detection, budgets, resumability, artifact inspection, comparison, export, and recovery for ordinary operators.

## Integrator

After every role returns, run one read-only integration step that:

1. builds a contradiction graph rather than averaging disagreement away;
2. preserves independent hypotheses;
3. constructs conservative, adjacent, and heretical candidate strata;
4. assigns each candidate a minimal falsification experiment;
5. requires matched-compute baselines and hidden evaluation;
6. ranks by expected information gain per cost without selecting a winner prematurely;
7. calculates path and resource leases;
8. emits a machine-readable experiment portfolio.

## Writer isolation

Only isolated writers with non-overlapping leases may implement experiments.

Shared orchestration, dependency, workflow, benchmark, authority, and evaluation files belong to one coordinator. Every writer must return implementation, focused tests, a reproducible command, machine-readable results, a resource receipt, and rollback instructions.

## Evolution loop

For each generation:

1. generate diverse hypotheses;
2. allocate cheap proxy experiments;
3. run hostile and contamination-resistant evaluation;
4. measure actual compute and memory;
5. diagnose mechanisms;
6. eliminate dominated candidates;
7. mutate or recombine mechanisms rather than names;
8. promote only effects reproduced across seeds and task families;
9. preserve negative results;
10. distill validated discoveries into the native model or learning system;
11. rerun broad regression suites;
12. stop at an exact receipt, never a persuasive narrative.

## Candidate genome

Every experiment must serialize its complete genome:

- model graph and tensor shapes;
- parameter-sharing rules;
- representation or tokenizer;
- state and memory update rules;
- routing or halting rules;
- learning objectives;
- data generators and mixtures;
- optimizer and schedule;
- initialization and precision;
- random seeds;
- inference budget;
- external tools with license and immutable pin;
- hardware;
- code revision;
- parent experiments.

Unserialized novelty does not count.

## Selection objective

Maintain a Pareto set over capability breadth, out-of-distribution generalization, factual reliability, calibration, adaptation, active compute, memory traffic, latency, training cost, reproducibility, and local deployability.

A candidate may be promoted only when its claimed advantage survives a matched-resource comparison, hidden adversarial evaluation, at least two seeds, broad regression, and a complete resource receipt.

## Anti-collapse rules

The runtime must not:

- converge immediately on the first plausible architecture;
- let one external model author the entire direction;
- use synthetic data without independent verification;
- treat chain-of-thought length as intelligence;
- hide retrieval, tool, cache, or repeated-inference costs;
- discard negative results;
- silently change evaluation sets;
- promote benchmark-specific patches into the general model;
- or call orchestration itself a new native model.

## Completion boundary

This phase is complete when a local operator can launch the foundry, state hardware and budget, run parallel research agents, receive a contradiction-aware portfolio, execute isolated proxy experiments, compare candidates under matched resources, inspect hidden evaluations and diagnoses, reproduce lineage, promote a validated candidate, export its model artifact, and receive an exact receipt distinguishing demonstrated gains from hypotheses.

The first milestone is not the final architecture. It is a functioning scientific machine capable of proving that an unexpected architecture or learning mechanism is better.
